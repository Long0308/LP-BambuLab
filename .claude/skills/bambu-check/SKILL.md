# Bambu A1 — Check & Recovery

Máy: **Bambu Lab A1 + AMS Lite**, kết nối **LAN MQTT (cổng 8883)**.
Chạy mọi lệnh dưới đây từ thư mục gốc của repo.

## 0. Nguyên tắc bất di bất dịch

**AI KHÔNG có quyền ra lệnh xuống máy in.** Mọi lệnh in / dừng / chỉnh nhiệt do **người dùng bấm trên web dashboard**. Tuyệt đối:

- ❌ **KHÔNG tạo file `.mcp.json`** ở thư mục gốc. Claude Code tự nạp project-scoped MCP từ file đó → trao AI toàn quyền (`start_print_job`, `set_temperature`, `delete_printer_file`…).
- ❌ Không gọi tool `mcp__bambu-printer__*` nào (đã bị disable trong `.claude/settings.json`).
- ✅ Cấu hình secret nằm ở `.env` (ưu tiên) hoặc `printer.local.json`. Cả hai đều gitignore.

## 1. Check nhanh trạng thái

```bash
python bambu_test.py          # one-shot: đọc 1 snapshot rồi thoát
```

Kết quả mong đợi: `=== OK - DOC DUOC MAY IN ===` kèm `gcode_state`, nozzle/bed, progress.

## 2. Nếu lỗi `AUTH_FAIL rc=5`

**Đây là lỗi thường gặp nhất.** Nghĩa: broker sống, mạng OK, chỉ **sai Access Code**.

> **Access Code TỰ ĐỔI** mỗi khi: bấm nút 🔄 trên máy, hoặc chuyển giữa **LAN Only ↔ Login Cloud**.

**Cách khôi phục tự động** — Bambu Studio lưu mã hiện tại theo serial:

```bash
# Đọc access code hiện tại (thay serial nếu khác)
grep -A2 '"access_code"' "$APPDATA/BambuStudio/BambuStudio.conf"
```

Kết quả dạng: `"access_code": { "<SERIAL_MAY_IN>": "xxxxxxxx" }`

Rồi cập nhật `.env` (chỉ sửa dòng `BAMBU_ACCESS_CODE`), hoặc:

```bash
python bambu_connect.py <IP> <SERIAL> <ACCESS_CODE>   # ghi printer.local.json + test cổng
```

**Lưu ý:** login cloud **KHÔNG** chặn LAN MQTT. Bạn dùng được **cả hai**:
cloud (Filament Library trừ gam theo RFID) + LAN dashboard (trạng thái realtime).

## 3. Chẩn đoán theo tầng

Chạy đúng thứ tự, dừng ở tầng đầu tiên fail:

| Tầng | Lệnh | Fail nghĩa là |
|---|---|---|
| Mạng | `Test-NetConnection <IP> -Port 8883` | PC/máy in khác mạng, hoặc firewall |
| Broker | cổng 8883 mở nhưng `rc=5` | Access Code sai → mục 2 |
| Auth OK | `bambu_test.py` in ra nhiệt độ | ✅ Xong |

Cổng: **8883** MQTTS (trạng thái/lệnh) · **990** FTPS (upload) · **6000** camera (1 người xem 1 lúc).

## 4. Chạy web dashboard

```bash
python bambu_web.py 8787
```

- PC: `http://localhost:8787`
- Điện thoại (cùng LAN): `http://<IP-PC>:8787`
- Lần đầu cần mở firewall (Admin): `Mo-Cong-Firewall-8787.bat`

Dashboard **read-only + nút điều khiển do người dùng bấm**. Server chỉ gửi lệnh khi có POST từ trình duyệt.

## 5. Audit bảo mật TRƯỚC KHI PUSH

Chạy đủ 4 lệnh, **tất cả phải sạch**:

```bash
# 1. Không được có file .mcp.json
ls -la .mcp.json 2>/dev/null && echo "NGUY HIEM: xoa ngay" || echo "OK"

# 2. .env và printer.local.json phải bị ignore
git check-ignore -v .env printer.local.json

# 3. Access code / serial KHÔNG được nằm trong cây commit
git grep -n "BAMBU_ACCESS_CODE=" HEAD ; git grep -nE "[0-9a-f]{8}" HEAD -- '*.json' '*.py'

# 4. Xem file nào sắp commit
git status --short
```

Chỉ được phép tracked: `.env.example`, `printer.local.example.json` (toàn `REPLACE_WITH_*`).

## 6. Sự thật cần nhớ về AMS (đừng báo sai cho người dùng)

- Field `remain` trong MQTT LAN của **A1/AMS Lite kẹt ở 100%** — **KHÔNG** phản ánh nhựa đã dùng.
- **KHÔNG** được tính `remain% × tray_weight` rồi báo "còn N gam" → sai.
- Bambu **có** theo dõi gam thật, nhưng ở **Filament Library trên cloud** (khai báo Net Weight + chip RFID, trừ dần sau mỗi job). Dữ liệu đó **không chảy qua MQTT LAN**.
- Muốn có gam trên dashboard: phải làm **tracker cục bộ** (khai báo Net Weight, khoá theo `tag_uid`, tự trừ theo job) hoặc gọi **Bambu Cloud API**.

## 7. Bẫy đã biết

- **OrcaSlicer 2.4.2** crash `Access violation (0xC0000005)` với model nặng + Variable Layer Height → **dùng Bambu Studio 2.7** (ổn định trên máy này).
- Bambu Studio báo *"configuration generated by a newer version… values replaced"* → **chỉ là thông báo**, bấm OK.
- Preset process **không xoá được Variable Layer Height** đã bake trong object → phải **Reset VLH** thủ công thì đổi `layer_height` mới có tác dụng.
